# チャットCMS Webアプリ 要件定義書（MVP）

元仕様書: `C:\Users\kento\Downloads\chat_cms_requirements.md`

## 1. 目的

WebやAIに詳しくない中小企業・小規模事業者でも、自然文のチャット操作でホームページを更新できるWebアプリを提供する。
本アプリは複雑な管理画面を前提とせず、ユーザーの曖昧な指示をAIが解釈し、候補提示・確認・反映までを支援する「超簡単チャットCMS」を目指す。
運営側は保守や更新代行を行わず、顧客は自ら更新を実行する。その代わり、誤更新対策として変更履歴・バージョン管理・ロールバック機能を中核機能とする。

## 2. スコープ

### 2.1 MVPで実現すること

- 顧客がログインして自社サイトを管理できる
- チャット形式で更新指示を出せる
- AIが指示内容を解釈し、候補案または変更内容を提示できる
- 顧客が候補を選択し、最終承認ボタンを押して反映できる
- 変更のたびにサイト全体状態をフルスナップショット保存できる
- 過去バージョンを一覧表示し、任意の版へロールバックできる
- 変更内容の要約と差分を履歴画面で確認できる
- テキスト変更、画像差し替え、お知らせ投稿ができる

### 2.2 MVPで実現しないこと

- 運営側の承認フロー
- 運営側の更新代行
- EC機能
- 会員機能
- 複雑なページビルダー
- 自由なレイアウト変更
- 高度なSEO運用機能
- 高度なアクセス解析ダッシュボード
- 問い合わせAIフィルタ
- 多言語自動生成

## 3. 想定ユーザー

### 3.1 主ユーザー

- WebやAIが苦手な中小企業経営者
- 社内にWeb担当者がいない小規模事業者
- 制作会社とのやり取りやCMS操作が苦手なユーザー

### 3.2 運営ユーザー

- 顧客管理・契約管理を行う管理者
- 保守や日々の更新対応は行わない

## 4. 提供価値

- 自然文で更新指示ができる
- どこをどう直せばよいか分からなくても進められる
- AIが候補を提示するため、文章が苦手でも更新できる
- 最終的な公開判断はユーザー自身が行える
- 間違っても必ず元に戻せる

## 5. システム構成

1. 公開サイト
2. 顧客向け管理画面（チャットCMS）
3. 運営向け管理画面（顧客・契約管理）

### 5.1 公開サイト

- 顧客ごとのホームページを表示する
- テンプレートベースで構築する
- 変更反映時に再生成する

### 5.2 顧客向け管理画面

- チャットUIを中心とする
- 更新指示、候補選択、承認、履歴確認、ロールバックを行う

### 5.3 運営向け管理画面

- 顧客一覧管理
- 契約状態管理
- サイト管理
- 操作ログ確認

## 6. 機能要件

### 6.1 認証・アカウント機能

#### 顧客機能

- メールアドレスとパスワードでログインできる
- 自社データのみにアクセスできる

#### 管理者機能

- 全顧客の情報を閲覧できる
- 顧客ごとのサイト情報を確認できる

### 6.2 顧客ダッシュボード

- 現在のサイト公開状態を確認できる
- 最近の更新履歴を確認できる
- チャット画面へ遷移できる
- お知らせ投稿画面へ遷移できる
- ロールバック履歴画面へ遷移できる

### 6.3 チャット更新機能

#### 概要

顧客は自然文で更新したい内容を入力できる。AIはその内容を解釈し、変更対象と変更候補または変更確認を提示する。

#### 対応操作（MVP）

- テキスト変更
- 画像差し替え
- お知らせ作成
- 問い合わせ先変更
- 営業時間変更

#### 会話フロー

1. 顧客が自然文で指示を入力する
2. AIが対象箇所を特定する
3. AIが候補または反映内容を提示する
4. 顧客が候補を選ぶ、または変更内容を確認する
5. 承認ボタン押下で反映する
6. 新バージョンを保存する

### 6.4 候補提示機能

- 曖昧な指示には候補提示を行う
- 明確な事実変更は即確認画面へ進めてもよい
- 候補は原則2〜3案、5案以上は出さない

### 6.5 承認機能

- 運営承認は不要
- 顧客自身の最終承認は必須
- 承認ボタン押下前はサイトに反映しない
- 承認操作はログに残す

### 6.6 テキスト変更機能

対象例:

- トップ見出し
- トップ説明文
- 会社紹介
- サービス説明
- 電話番号
- メールアドレス
- 営業時間

### 6.7 画像差し替え機能

- 顧客が画像をアップロードできる
- 差し替え対象を特定できる
- プレビュー表示ができる
- 画像は自動リサイズ・最適化する
- 利用箇所ごとに比率制約を持つ

### 6.8 お知らせ投稿機能

- タイトル、本文、画像1枚を入力して投稿できる
- AIにより本文整形、見出し調整、改行調整ができる
- 公開前にプレビュー確認できる
- 投稿後は新バージョンとして保存する

### 6.9 バージョン管理機能

変更確定のたびに、サイト全体状態をフルスナップショットとして保存する。

保存対象:

- 全ページコンテンツ
- 画像参照情報
- ナビゲーション
- テーマ設定
- お知らせ表示状態
- 問い合わせ先情報
- テンプレート情報
- スキーマバージョン

### 6.10 ロールバック機能

- バージョン一覧から任意の版を選択できる
- 変更概要を見て戻す対象を選べる
- 変更前後または概要を確認できる
- 「この版に戻す」ボタンでロールバックできる
- ロールバック結果も新しいバージョンとして保存する

### 6.11 差分表示機能

- 各バージョンに対して変更点を要約表示する
- 変更前後の値を表示できる
- 対象ページ・対象箇所を表示できる

### 6.12 テンプレート管理機能

- サイトはテンプレートベースで作成する
- MVPではテンプレート数を限定する
- テンプレートごとに編集可能フィールドを定義する
- 自由なレイアウト編集は許可しない

### 6.13 運営向け顧客管理機能

- 顧客一覧表示
- 契約プラン確認
- 顧客ごとのドメイン情報確認
- 顧客ごとのサイト情報確認
- ログインユーザー確認
- 操作ログ確認

## 7. AI要件

### 7.1 AIの役割

- 自然文指示の意図解釈
- 対象ページ・対象セクション・対象フィールドの特定
- 文言候補の生成
- お知らせ本文の整形
- 変更概要の要約生成

### 7.2 AIの制約

- AIの出力をそのままHTMLへ反映しない
- AI出力は必ず構造化データへ変換する
- 実行可能性の最終判定はアプリケーション側が行う
- 許可されていない操作は拒否または追加質問する

### 7.3 マスタープロンプト要件

- 顧客ごとのサイト構造を理解させる
- 許可された変更範囲のみ出力させる
- JSONなどの構造化形式で出力させる
- 曖昧な依頼には候補提案を優先する
- 明確な事実変更には即確認を優先する

## 8. 非機能要件

### 8.1 可用性

- 顧客が日常的に利用できる安定性を持つ
- 更新処理失敗時は反映しない

### 8.2 安全性

- 顧客ごとのデータ分離を行う
- 不正アクセス防止
- 操作ログ保存
- 画像アップロード制限
- バリデーション実施

### 8.3 復元性

- 変更のたびにフルスナップショット保存
- ロールバックを1操作で実行可能
- ロールバック後も履歴を保持

### 8.4 性能

- 小規模サイト向けMVPとして十分な性能を持つ
- 初期はフルスナップショット方式を採用し、差分最適化は将来検討とする

- `schemaVersion` と `templateVersion` を必ず含める

## 12. 受け入れ条件（MVP）

1. 顧客がログインできる
2. 顧客がチャットで更新指示を出せる
3. AIが候補または変更内容を提示できる
4. 顧客が承認ボタンで変更反映できる
5. 反映ごとに新バージョンが作成される
6. 過去バージョン一覧を確認できる
7. 任意の版にロールバックできる
8. ロールバック後も新しい版として履歴に残る
9. テキスト変更・画像差し替え・お知らせ投稿ができる
10. 変更点要約が履歴画面に表示される

### 画面設計原則

- ボタン数を増やしすぎない
- 専門用語を避ける
- チャットだけに依存しすぎず、確認と候補選択をUIで補助する
- 本番反映の前には必ず確認ステップを置く
- ロールバック導線は常に分かりやすくする
- 変更履歴は日時だけでなく内容も見せる
- 管理画面はスマホでも最低限利用できることを前提とする

## 14. DBスキーマ詳細設計

### 設計方針

1. 顧客ごとのデータ分離を前提とする
2. 公開状態の正本は `site_versions.snapshot_json` とする
3. HTMLはレンダリング結果であり、正本にしない
4. 変更のたびにフルスナップショットを保存する
5. 差分は復元用ではなく表示用として保持する
6. チャット上の曖昧な会話状態と、確定済み変更状態を分離する

### 主な関係

- clients 1 : N users
- clients 1 : N sites
- site_templates 1 : N sites
- sites 1 : N site_versions
- sites 1 : N assets
- sites 1 : N news_posts
- sites 1 : N chat_sessions
- sites 1 : N change_sets
- site_versions 1 : N version_changes
- chat_sessions 1 : N chat_messages
- chat_sessions 1 : N suggestion_sets

## 15. マスタープロンプト仕様

### AIの役割定義

AIが行うこと:

1. 顧客の自然文を解釈する
2. 変更対象を特定する
3. 必要に応じて候補を2〜3案生成する
4. 反映前の確認文を生成する
5. JSON形式の構造化出力を返す

AIが行わないこと:

1. 直接HTMLを生成して公開すること
2. 許可されていない構造変更を勝手に実行すること
3. システム側の検証を飛ばして反映すること
4. 不明瞭な指示を推測で即公開すること

### 指示分類ルール

- expression_update
- factual_update
- asset_update
- content_create
- unsupported_request

### 出力形式

AIの内部出力は必ずJSON形式にする。

必須項目:

- action_type
- intent_category
- confidence
- target_page
- target_section
- target_field
- needs_confirmation
- needs_more_input
- user_message
- assistant_message

## 16. 実装設計

### 実装方針

- フロントエンドとバックエンドは Next.js で単一リポジトリ構成にする
- DB / Auth / Storage は Supabase を使う
- AI連携は OpenAI API を使う
- デザインシステムは最小限に抑え、機能優先で進める
- テンプレートは1種類から始める
- 公開反映は承認後のみ行う
- フルスナップショット版管理を最初から入れる

### 推奨リポジトリ構成

- app/
- components/
- lib/
- prompts/
- supabase/
- types/
- tests/
- docs/

### 実装優先順位

1. 土台
2. 公開サイト表示
3. 顧客ダッシュボード
4. チャット更新MVP
5. ロールバック
6. お知らせと画像
7. 運営画面
